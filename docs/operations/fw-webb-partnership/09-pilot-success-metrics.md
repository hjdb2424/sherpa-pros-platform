# FW Webb Pilot — 12-Month Success Metrics + Performance Gates

## Purpose

Define the **performance gates** that govern the 12-month pilot. These gates determine the post-pilot expansion decision per LOI §9 and feed the Hub Manager + Network Ops dashboards (specs 06 + 07).

## Gate structure

Three layers, each with explicit thresholds:

1. **Per-Hub gates** — every Pilot Branch evaluated independently
2. **Network gates** — across all 5 Pilot Branches in aggregate
3. **FW Webb-side gates** — incremental impact on FW Webb branches (validates Slide 9 economic model)

Each gate has a **Pilot Month 6** ("M6") interim checkpoint and a **Pilot Month 12** ("M12") final checkpoint.

## Gate scoring methodology

Each individual gate scored:

- **Met or exceeded** = 100%
- **At or above floor** = 70%
- **Below floor** = 0%

Per-layer aggregate score = average across all gates in that layer.

Pilot exit score = weighted average across layers:

- 50% Per-Hub gates (Layer 1)
- 30% Network gates (Layer 2)
- 20% FW Webb-side gates (Layer 3)

## Pilot exit decisions (per LOI §9)

Pilot exit score determines post-pilot path:

| Pilot exit score | Decision |
|---|---|
| **>= 95%** | **Best-case** — expand to 25 Hubs Year 2 + accelerate to 50+ Hubs Year 3 |
| **80% - 94%** | **Plan-case** — expand to 25 Hubs Year 2 |
| **70% - 79%** | **Selective expansion** — replicate the 3 best-performing Hubs; pause on the lower 2 |
| **50% - 69%** | **Pilot extension** — 6-month extension at revised gates with corrective actions |
| **< 50%** | **Pilot wind-down** with FW Webb |

## Per-Hub override

Even if the network exit score is >= 80%, **individual Pilot Branches that fail their own per-Hub gates** are reviewed independently. A failing Hub may be:

- Closed at Pilot end
- Relocated within the same FW Webb branch
- Substituted with an alternate FW Webb branch
- Reformatted (e.g. smaller footprint, no member lounge)

This protects the partnership from a single-Hub failure dragging down the network-wide narrative.

## Review cadence

| Cadence | Owner | Audience | Output |
|---|---|---|---|
| Daily | Hub Manager | Hub Manager | Per-Hub dashboard refresh; alerts triage |
| Weekly | Hub Manager → Regional VP | Sherpa Network Ops + FW Webb regional VP | Weekly metrics rollup; alerts escalation |
| Monthly | Network Ops | Sherpa Pros leadership | Monthly P&L close; network rollup; gate progress |
| Quarterly | Joint executive sponsors | Sherpa Pros + FW Webb leadership | **QBR** — formal gate review + corrective actions |
| Pilot M6 | Joint executive sponsors | Both Parties leadership | **M6 interim** — go/no-go on Pilot continuation |
| Pilot M12 | Joint executive sponsors | Both Parties + counsel | **Pilot exit** — expansion decision per LOI §9 |

## Data sources

All gate metrics computed from:

- `members`, `member_visits`, `member_nps_responses` (Sherpa Pros DB)
- `materials_orders`, `hub_materials_staging`, `hub_returns` (Sherpa Pros DB)
- `hub_metrics_daily` (computed nightly rollup — see spec 05 §8)
- FW Webb monthly Pilot-Branch data feed (LOI §6.3) — manual or EDI depending on integration phase
- FW Webb control-branch data (selected during LOI diligence) for vs-control comparisons

## Risks to gate achievement (and mitigation)

| Risk | Mitigation |
|---|---|
| Hub-buildout delay pushes M6 milestone | Stagger Hub openings; M6 measured M+6 from each Hub's individual open date for per-Hub gates |
| Member acquisition slower than plan | FW Webb co-marketing intensification; founder Phyrom on-site for member onboarding events |
| Materials orchestration lower than plan | Expand Top-200 SKU list per Hub; reduce Buffer→Counter-pickup ratio if buffer SKUs underperform |
| FW Webb data feed delayed (no vs-control compare) | Use Sherpa-attributed member spend as proxy; reconstruct from FW Webb POS at QBR |
| One Pilot Branch dragging network gates | Per-Hub override clause (see above) |
